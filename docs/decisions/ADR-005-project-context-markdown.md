# ADR-005: Markdown-Based Project Context (Not JSON Config)

## Status
ACCEPTED

## Date
2026-04-09

## Context
Agents need shared technical standards (stack, rules, conventions) but the format must be both human-editable and machine-parseable. Inspired by BMAD-METHOD's project-context.md pattern.

## Decision
`.flow/project-context.md` is a Markdown file with H2 sections and optional yaml code blocks. Parsed by `project_context.rs` using regex section splitting. Auto-generated by `flowctl init` with detected stack info.

## Consequences
- **Constraint**: Sections must follow exact heading names (Technology Stack, Guard Commands, etc.)
- **Benefit**: Human-readable AND git-diffable (vs opaque JSON/TOML)
- **Benefit**: Users edit with any text editor, no special tooling needed
- **Trade-off**: Regex parsing is fragile if users deviate from format. Mitigated by structured yaml blocks for machine-critical data (Guard Commands, File Conventions)

## Rejected Alternatives
- JSON config: Not human-friendly, hard to add comments/rationale
- TOML: Better than JSON but still config-oriented, not documentation-oriented
- YAML-only: Loses the prose sections (Architecture Decisions, Non-Goals)
